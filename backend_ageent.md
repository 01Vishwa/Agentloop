# Backend & AI Agent System Review

## Audit Findings

| Component | Issue | Impact | Recommended Fix |
| :--- | :--- | :--- | :--- |
| **DS-STAR Orchestrator** | - Inner execution loops strictly bounded by `max_debugger_retries`.<br>- Outer loop bounded by `max_rounds`.<br>- Token limits correctly abort execution.<br>- If Verifier fails, it defaults to `is_sufficient=False` (fail-closed).<br>- *Minor Issue*: If RouterAgent throws an exception, the orchestrator logs a warning and continues to the next round with the same plan, potentially causing redundant execution. | Low | - The orchestrator design is mostly robust. To fix the minor issue, abort the current orchestrator run or retry the router if it fails completely. |
| **Agent Modules** | - Planner, Coder, Verifier, and Router use `.with_structured_output()`. Models lacking function-calling (e.g., `google/gemma-4-31b-it`, `mistralai/mixtral-8x7b-instruct-v0.1`) are safely handled via `json_mode` fallback in `llm_client.py`.<br>- `DsStarOrchestrator` implements `_with_retry` (3 attempts) for core agents.<br>- *Issue*: `SubQuestionGeneratorAgent` and `ReportWriterAgent` in the Deep Research Orchestrator lack retry logic for LLM failures. | Medium | - Apply `tenacity.AsyncRetrying` (the `_with_retry` helper) to `SubQuestionGeneratorAgent` and `ReportWriterAgent` calls to protect long-running research tasks from failing at the last step. |
| **Deep Research Orchestrator** | - The `asyncio.Semaphore(3)` is properly scoped using an `async with` block, preventing semaphore leaks even on agent crashes.<br>- `ReportWriterAgent` explicitly instructs the LLM to cite source sub-questions `[Q1]`, `[Q2]`. No groundedness gap found. | None | - Implementation is correct and well-scoped. No immediate fixes required. |
| **Code Executor** | - `subprocess.run(capture_output=True)` captures stdout/stderr entirely in memory. A runaway `print()` loop in the sandboxed code will exhaust memory (OOM) before `EXECUTION_TIMEOUT_SECONDS` triggers.<br>- *Artifacts*: `_collect_artifacts` correctly collects all files in the execution `tmpdir` (not just `./outputs/`). However, if code writes to absolute paths like `/tmp`, they are silently lost. | High (DoS risk) | - Replace `subprocess.run(capture_output=True)` with `subprocess.Popen`, reading streams with a byte-size limit (e.g., 2MB) to prevent OOM.<br>- Run code in a chroot/jail or strictly override `TMPDIR` to prevent writing outside the sandbox. |
| **LLM Client** | - API keys are loaded via the environment.<br>- *Issue*: There is no fallback if the primary provider (NVIDIA NIM) experiences downtime or rate limits. | High | - Implement a secondary LLM provider fallback (e.g., OpenAI, Anthropic, or Azure) in `get_nim_llm()` to ensure high availability. |
